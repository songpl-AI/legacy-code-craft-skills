# 分析质量标准

## 状态标记原则

- `🆕 全新`：代码库中完全没有，从零实现
- `🔧 改造`：现有脚本/接口需要修改或扩展
- `✅ 已有`：代码已实现，无需改动（仅作说明）

## 工作量估算原则

- 半天以内（< 0.5 天）：改几行、加一个字段、加一个 Watch
- 短期（0.5 ~ 2 天）：新增一个完整功能点或改造一个现有函数
- 中期（2 ~ 5 天）：新增一个完整脚本或多个功能点联动
- 长期（5+ 天）：涉及多个脚本改造，或依赖多个新接口

## 客户端 vs 服务器判断

- 客户端：纯 UI 显示、动画、本地状态切换
- 服务器：数据存储、跨用户交互、排行榜、奖励发放、防作弊

## 风险评估原则

- 修改高频函数（initialize / RefreshAll）→ 🔴 高
- 修改现有逻辑（非追加）→ 至少 🟡 中
- 纯新增函数/脚本 → 🟢 低

## 疑问清单原则

- 宁可多问，不要假设
- 特别关注：数值边界、周期重置、多端同步、跨用户场景
- `待确认问题.md` 用非技术语言描述，策划能看懂
