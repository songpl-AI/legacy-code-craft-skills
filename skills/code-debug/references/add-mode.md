# 模式1：添加调试日志

## Step 1 — 读取并分析脚本

1. 完整读取目标脚本
2. 识别所有函数、HTTP 请求、Watch/Fire 事件、状态切换点
3. 如果用户指定了关注的函数/功能，只在相关代码中添加
4. 如果未指定，在所有关键路径添加

## Step 2 — 幂等检查

**插入前必须先执行**：
1. 搜索脚本中已有的 `g_LogError("[DEBUG]` 行
2. 如果已存在 [DEBUG] 日志，提示用户：
   ```
   ⚠️ 脚本中已有 N 条 [DEBUG] 日志，建议先执行 `/code-debug clean` 清理后再重新插入，避免重复叠加。
   继续插入将跳过已有 [DEBUG] 的位置。
   ```
3. 插入时跳过已有 [DEBUG] 日志的相邻行（同函数内已有同类 [DEBUG] 则不再重复插入）

## Step 3 — 插入规则

**通用规则**：
1. 所有调试日志必须带 `[DEBUG]` 前缀，格式为 `g_LogError("[DEBUG][函数名] 描述" .. tostring(变量))`
2. 只新增代码，不修改已有逻辑
3. 变量拼接必须用 `tostring()` 包裹，防止 nil 拼接报错
4. 字符串连接用 `..`，不能用 `+`
5. 如果变量可能是 table，用 `tostring()` 即可（不需要 table.dump，避免日志太长）

**循环内日志控制**：
- 如果循环次数 <= 10：可以每次都打印
- 如果循环次数 > 10：只在 `i == 1` 或异常条件时打印
- 示例：
  ```lua
  for i = 1, count do
      if i == 1 or currentPoints >= thresholds[i] then
          g_LogError("[DEBUG][函数名] loop i=" .. tostring(i) .. " val=" .. tostring(val))
      end
  end
  ```

## Step 4 — 输出报告

```
## 调试日志报告

> **报告类型**：调试日志插入 | **生成时间**：YYYY-MM-DD

**脚本**: 文件名
**插入数量**: N 条 [DEBUG] 日志
**覆盖函数**: 函数1, 函数2, ...

### 插入明细

| # | 行号 | 函数 | 日志内容摘要 |
|---|------|------|-------------|
| 1 | ~100 | OnXxx | 函数入口，参数 xxx |
| 2 | ~105 | OnXxx | HTTP 请求参数 |
| ... | | | |

### 调试完成后清除
执行 `/code-debug clean 脚本路径` 可一键移除所有 [DEBUG] 日志
```
