# 归档最佳实践

## 何时创建归档？

✅ **应该归档**：
- 新增了完整功能模块
- 修改了核心业务逻辑
- 实现了复杂的交互流程
- 团队认为值得沉淀的代码

❌ **不需要归档**：
- 简单的 UI 调整（改个颜色、位置）
- 单行代码修复
- 临时的调试代码
- 即将删除的功能

---

## 归档粒度建议

| 粒度 | 说明 | 示例 |
|------|------|------|
| ✅ 合适 | 一个完整的用户可感知功能 | "助力功能"、"点赞功能" |
| ⚠️ 过大 | 多个独立功能混在一起 | "所有功能" |
| ⚠️ 过小 | 单个函数或细节 | "刷新按钮点击" |

---

## 归档命名建议

- ✅ 简短清晰："助力功能"、"点赞功能"
- ✅ 用户视角："学伴筛选"、"快速采摘"
- ❌ 技术术语："OnClick 实现"
- ❌ 过于笼统："新增代码"

---

## 注意事项

1. **归档不替代代码注释**：归档是宏观总结，注释是细节说明
2. **归档需要定期更新**：功能修改后记得更新归档
3. **归档是团队资产**：可以分享给其他开发者
4. **归档文件可搜索**：使用 Grep 在归档中搜索关键词

---

## 与其他 Skills 的配合

### code-modify 完成后
```bash
# 1. 修改代码
/code-modify scripts/example.lua 添加功能

# 2. 修改完成后，会提示：
# "是否需要归档？回复'归档'或功能名称"

# 3. 用户回复：功能名称

# 4. 自动执行归档
/code-archive create scripts/example.lua 功能名称
```

### code-analyze 检查归档
```bash
# 1. 分析脚本
/code-analyze scripts/example.lua

# 2. 如果发现已有归档，会先提示：
# "📚 该脚本已有 2 个功能归档，是否查看？"

# 3. 用户可以选择先查看归档，再分析
```
