# Step 6-7: 执行修复 & 收尾

## Step 6 — 执行修复（需用户确认）

询问用户：
```
是否按推荐方案执行修复？（将走 code-modify 标准流程：备份 → 修改 → 自检）
(y/n)
```

**用户确认后**，自动执行 code-modify 流程：
1. 备份原文件到 `scriptsBackup/`
2. 按推荐方案修改代码
3. 输出修改说明

---

## Step 7 — 收尾

### 文件模式（传入了 Bug 文件路径）

在 Bug 文件的 frontmatter 中更新：
- `status: pending` → `status: fixed`
- 新增 `fixed_time: YYYY-MM-DD`
- 新增 `fix_summary: 一句话描述修复内容`

### 粘贴模式（直接粘贴 Bug 内容）

无本地文件可更新，提示开发手动处理：
```
✅ 修复完成！
请手动在 Bug 系统中：
1. 将任务状态标记为「已修复/待验证」
2. 在备注中追加：已修复 - {一句话描述修复内容}
```

### 两种模式都执行

在对应归档的修改历史中追加一条记录：
```
| YYYY-MM-DD | 开发者 | 修复 Bug: BUG-xxx - {bug标题} |
```
