# Step 3-5: 日志分析 & 代码定位 & Bug 报告

## Step 3 — 日志分析

**如果有日志扫描摘要**：直接读取并引用，不重复扫描。

**如果有日志原文但无摘要**：
1. 从日志中提取关键信息：错误信息、函数名、code 值、nil 异常等
2. 结合 Step 2 的函数列表，缩小代码定位范围
3. 输出日志分析结论：
```
## 日志分析

- 日志中出现 `[InitUI] ERROR` → 初始化阶段出错
- 未见 `[OnEvent]` 日志 → 说明按钮点击事件未触发
- 可能原因：节点路径找不到导致按钮未绑定
```

**如果没有日志**：
```
⚠️ Bug 报告中没有日志，定位精度会下降。
建议让测试开启真机日志后重新复现，搜索关键字：[TAG名]
```

---

## Step 4 — 读取相关代码

1. 根据前几步定位的脚本和函数，读取相关代码段
2. 结合 Bug 描述的操作路径，追踪代码执行链
3. 定位最可能的 bug 位置

---

## Step 5 — 输出 Bug 分析报告

```
## Bug 分析报告

> **报告类型**：Bug 分析 | **生成时间**：YYYY-MM-DD

### Bug 基本信息
- Bug ID：BUG-xxx
- 功能模块：{TAG} - {功能名}
- 严重程度：{severity}

### 关联代码
- 脚本：scripts/script_name.lua
- 最可能的 bug 位置：函数名（第 XXX 行）

### 根因分析
[描述 bug 的根本原因]

### 影响范围
- 直接影响：[受影响的功能]
- 关联事件：[可能影响的 Watch/Fire]

### 修复方案
[描述修复思路，不超过 3 种方案时列出对比]

### 推荐修复
[推荐的修复方式及理由]
```
