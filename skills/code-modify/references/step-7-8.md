# Step 7-8: 更新索引 + 提示归档

## Step 7 — 强制更新脚本索引（必须执行）

修改完成后，**无论是否归档，都必须确保脚本已在 script-index.md 中**：

1. 读取 `{AGENT_DIR}/{baseDir}/script-index.md`
2. 检查当前脚本的 TAG 是否已存在
3. **已存在** → 更新「脚本文件」列（文件名可能变更）
4. **不存在** → 在 `<!-- INDEX_END -->` 前插入新条目

格式：`| TAG | 脚本文件名前缀（前8位） | 功能定位 | 关键功能（本次修改涉及的，逗号分隔） | 核心接口 |`

> 这一步保证即使开发没有归档，code-bugfix 也能通过 script-index 找到脚本文件。

---

## Step 8 — 提示归档（在输出末尾）

修改完成后，根据修改类型判断是否建议归档：

**必须归档**（在输出末尾强烈建议）：
- 新增完整功能模块（新函数 + 新接口 + 新 UI 交互）
- 联调新接口（通过 API Doc URL 联调的）
- 跨脚本事件改动（新增或修改 Watch/Fire 事件）
- 核心业务流程变更

**可选归档**（在输出末尾提示）：
- 单个函数的小功能添加
- 已有功能的增强

**不需要归档**（不提示）：
- UI 微调（改颜色、位置、文案）
- 单行 bug 修复
- 日志调整

提示格式：
```markdown
---

💡 本次修改[新增了完整功能/联调了新接口/...]，建议归档: `/code-archive create 脚本路径 功能名称`
```

---

## 处理复杂修改

如果修改范围很大（风险等级=高）：
1. 建议拆分为多个小步骤
2. 每个步骤独立备份、修改、验证
3. 逐步推进，降低风险

---

## 遇到不确定的情况

如果遇到以下情况，必须停下来询问用户：
- 不清楚某个变量的含义或来源
- 不确定某段代码的业务逻辑
- 有多种实现方式，不确定用户期望哪种
- 修改可能影响其他功能，但无法确认影响范围
- 发现代码中有明显问题，不确定是否应该顺便修复

**切记：不要猜测，不要假设，遇到不确定的地方一定要问！**
