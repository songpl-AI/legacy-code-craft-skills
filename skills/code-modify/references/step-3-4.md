# Step 3-4: 备份 + 编写代码

## Step 3 — 备份原文件

修改前自动备份原文件：
```bash
cp 原文件路径 scriptsBackup/文件名_$(date +%s).lua
```

备份后告知用户备份路径。

---

## Step 4 — 编写代码

遵循以下原则：

### 修改策略（严格版）

**核心原则**：**只新增，不修改旧代码**（除非用户明确要求）

- **✅ 优先新增**：新功能用新函数、新事件、新 UI 节点实现
- **❌ 禁止修改**：任何已有代码（包括 bug 修复、重构）
- **⚠️ 例外情况**：
  - 用户明确说"修改 XXX 函数"
  - 用户明确说"重构 XXX"
  - 用户明确说"修复 XXX bug"

**风险报告中强制说明**：
- 是否修改了已有代码：[是/否]
- 如果"是"：
  - 修改位置：[列出所有函数和行号]
  - 修改原因：[用户明确要求 / AI 判断需要]
  - ⚠️ **如果是 AI 判断，必须再次确认用户是否同意**

### 代码规范

1. 添加 TAG 标识：`local TAG = "[模块名]"`
2. 关键路径添加日志：
   - 普通日志：`g_Log(TAG, "[函数名] 描述", variable)`
   - 真机调试：`g_LogError("[函数名] 描述" .. tostring(variable))`
3. HTTP 请求必须有 fail 回调
4. Watch 参数用 `args[0]`
5. 字符串连接用 `..`，数值要 `tostring()`
6. 添加必要注释说明修改目的
7. Exit() 中添加清理逻辑（协程/定时器/GameObject）
